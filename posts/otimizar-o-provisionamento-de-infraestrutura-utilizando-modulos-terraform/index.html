Otimizar o provisionamento de recursos de infraestrutura utilizando módulos Terraform

janeiro 25, 2025 · 4 minutos · 735 palavras · Roberto Santos

Utilizar as práticas de desenvolvimento de software no provisionamento e gerenciamento de infraestruturas tecnológicas, é um dos principais pilares da Infraestrutura como Código - IaC. Isso nos permite criar componentes consistentes, testáveis e reutilizáveis.

Em uma linguagem de programação de uso geral, assim como o Python, se você tem o mesmo trecho de código escrito em diversas partes do seu projeto, você pode colocá-lo dentro de uma função e reutilizar em todos os locais:

# Definindo a função em um lugar
def hello():
    print ("Hello World!")


# Utilizando esta função em multiplos locais
hello()

De maneira similar, no Terraform temos o conceito de módulo, que consiste em uma coleção de arquivos .tf e/ ou .tf.json agrupados em um diretório. Desta maneira você pode agrupar os seus arquivos dentro de um módulo terraform e reutilizar em vários locais.

Os módulos terraform podem ser classificados como módulo root ( root module ) ou módulo filho ( child module ).

O módulo root é o diretório principal, onde são definidos os recursos do terraform, e, onde você executa comandos como terraform init e/ou terraform apply.

Um módulo Terraform (geralmente o módulo root de uma configuração) pode chamar outros módulos para incluir seus recursos na configuração. Um módulo que foi chamado por outro módulo é frequentemente chamado de módulo filho.

module "modulo_filho" {  
 source = "../modulo_root"  
 ... 
}

Em um projeto onde você tem, por exemplo, os ambientes de staging e produção, em vez de ter o mesmo código copiado e colado em várias pastas, você criará os dois ambientes reutilizando o código do mesmo módulo root, alterando os inputs de acordo com cada ambiente.

module "staging" {
    source= "path-to-module-example"
    ...
}

module "production" {
    source= "path-to-module-example"
    ...
}

Para mais informações sobre os módulos terraform, recomendo a leitura da documentação oficial.

Visão Geral#

Neste guia, criaremos nosso módulo refatorando os artefatos deste repositório, que foram utilizados para demonstração do post Gerenciando recursos do Proxmox com o Terraform.

Criando o módulo#

Vamos clonar o repositório, e, para um melhor entendimento, vamos chamá-lo de “modulo_root”:

git clone https://github.com/BarryL4bs/terraform-bpg-proxmox.git modulo_root

No arquivo variables.tf vamos adicionar nossas variáveis de entrada ( inputs ), suas possíveis validações e/ou valores default.


# Variáveis do resource proxmox_virtual_environment_role

variable "role_id" {
  description = "The role identifier"
  type        = string
}

variable "privileges" {
  description = "The role privileges"
  type        = list(string)
}

# Variáveis do resource proxmox_virtual_environment_user

variable "acl_path" {
  description = "The path where the access control list will be applied"
  type        = string
}

variable "acl_propagate" {
  description = "Whether to propagate to child paths"
  type        = bool
}

variable "user_id" {
  description = "The user identifier"
  type        = string
}

# Variáveis do resource proxmox_virtual_environment_user_token

variable "privileges_separation" {
  description = "Restrict API token privileges with separate ACLs (default), or give full privileges of corresponding user"
  type        = bool
  default     = false
}

variable "expiration_date" {
  description = "Expiration date for the token"
  type        = string
  default     = null
}

variable "token_name" {
  description = "User-specific token identifier"
  type        = string
}

Definidas as nossas variáveis, faremos duas modificações no arquivo main.tf:

  1. Substituir os valores fixos que estão inseridos diretamente no código (também conhecidos como hardcoded), pelas variáveis recém-criadas;
  2. Alterar o nome dos recursos por um termo genérico (aqui chamaremos de “this”).
resource "proxmox_virtual_environment_role" "this" {

  role_id    = var.role_id
  privileges = var.privileges
}

resource "proxmox_virtual_environment_user" "this" {

  acl {
    path      = var.acl_path
    propagate = var.acl_propagate
    role_id   = var.role_id
  }

  user_id = var.user_id

  depends_on = [
    proxmox_virtual_environment_role.this
  ]
}

resource "proxmox_virtual_environment_user_token" "this" {

  expiration_date       = var.expiration_date
  privileges_separation = var.privileges_separation
  token_name            = var.token_name
  user_id               = var.user_id

  depends_on = [
    proxmox_virtual_environment_user.this
  ]
}

No arquivo outputs.tf altere o nome dos recursos, assim como fizemos no arquivo main.tf.

output "id" {
  value = proxmox_virtual_environment_user_token.this.id
}

output "api-token" {
  value     = proxmox_virtual_environment_user_token.this.value
  sensitive = true
}

Criando recursos utilizando o módulo#

Criaremos um diretório e nele seus um arquivo de configuração .tf que irá chamar outro módulo.

Chamar um módulo significa incluir o conteúdo desse módulo na configuração com valores específicos para suas variáveis ​​de entrada. Os módulos são chamados de dentro de outros módulos usando blocos de módulos.

module "modulo_filho" {
  source = "../modulo_root"

  role_id = "TF-Module-Role"

  privileges = [
    "Datastore.Allocate", "Datastore.AllocateSpace", "Datastore.AllocateTemplate", "Datastore.Audit", "Pool.Allocate", "Sys.Audit", "Sys.Console", "Sys.Modify", "SDN.Use", "VM.Allocate", "VM.Audit", "VM.Clone", "VM.Config.CDROM", "VM.Config.Cloudinit", "VM.Config.CPU", "VM.Config.Disk", "VM.Config.HWType", "VM.Config.Memory", "VM.Config.Network", "VM.Config.Options", "VM.Migrate", "VM.Monitor", "VM.PowerMgmt", "User.Modify",
  ]
  acl_path      = "/"
  acl_propagate = true

  user_id = "TF-Module-User"

  token_name = "TF-Module-Token"
}
